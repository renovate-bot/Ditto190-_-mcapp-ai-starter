# Communication Channels - {{projectName}}

This document outlines how to communicate with the {{projectName}} team for various purposes.

## 📥 Incoming Communication (How to Reach Us)

### 🐛 Bug Reports

**Channel**: [GitHub Issues](https://github.com/{{githubOrg}}/{{projectName}}/issues)  
**Process**:

1. Search existing issues first
2. Use bug report template if available
3. Include detailed reproduction steps
4. Provide environment information (VS Code version, OS, etc.)

**Response SLA**: 2 business days  
**Escalation**: Tag @trusted-committers for urgent issues

### 💬 Pull Request Questions

**Channel**: PR comments on your pull request  
**Process**:

- Tag specific reviewers with `@username`
- Use `@trusted-committers` for general questions
- Include context and specific questions

**Response SLA**: 1 business day during review  
**Escalation**: Slack channel `#{{projectName}}-dev` (if available)

### 💡 Feature Requests

**Channel**: [GitHub Discussions](https://github.com/{{githubOrg}}/{{projectName}}/discussions)  
**Process**:

1. Check existing discussions for similar requests
2. Use feature request template if available
3. Describe use case and expected value
4. Consider contributing if you have development resources

**Response SLA**: 5 business days  
**Escalation**: Contact Team Lead directly

### 📚 Documentation Questions

**Channel**: GitHub Issues with `documentation` label  
**Process**:

- Be specific about which documentation section
- Suggest improvements if possible
- Include examples of what's unclear

**Response SLA**: 3 business days  
**Escalation**: Create PR with documentation improvements

### 🔒 Security Issues

**Channel**: Private communication required  
**Process**:

1. **DO NOT** report security issues in public
2. Email: {{internalContact}}
3. Include "Security: {{projectName}}" in subject
4. Provide detailed vulnerability description

**Response SLA**: 1 business day  
**Escalation**: Security team immediately

## 📤 Outgoing Communication (How We Inform Others)

### 🚀 Feature Releases

**Channel**: GitHub Releases  
**Frequency**: As needed (typically monthly)  
**Content**:

- New features and improvements
- Breaking changes (clearly marked)
- Migration instructions
- Link to updated documentation

**Advance Notice**: 1 week for breaking changes

### ⚠️ Breaking Changes

**Channel**:

- GitHub Issues (breaking change label)
- Email notification to registered users
- Slack announcement in relevant channels

**Timeline**:

- **Announcement**: 2 weeks before change
- **Beta available**: 1 week before change
- **Final release**: As scheduled

### 🔄 Planned Maintenance

**Channel**: GitHub Status (if configured)  
**Frequency**: As needed  
**Content**:

- Scheduled downtime
- Service interruptions
- Migration activities

**Advance Notice**: 3 business days when possible

### 📊 Roadmap Updates

**Channel**: GitHub Discussions or Wiki  
**Frequency**: Quarterly  
**Content**:

- Planned features for next quarter
- Priority changes
- Community feedback incorporation

### 🎉 Community Highlights

**Channel**: GitHub Discussions  
**Frequency**: Monthly  
**Content**:

- Notable contributions
- New trusted committers
- Usage statistics
- Community achievements

- **Purpose**: Progress review, blocker discussion, planning

## 📋 Communication Guidelines

### Best Practices

- **Be specific and clear** in all communications
- **Include context** when asking questions
- **Use appropriate channels** for different types of requests
- **Follow up** if you don't receive response within SLA
- **Be respectful** of everyone's time and expertise

### Response Time Expectations

- **Acknowledgment**: Within 1 business day
- **Full Response**: Within SLA for each channel
- **Urgent Issues**: Same day when possible
- **Weekends/Holidays**: Best effort basis

### Documentation Updates

All significant decisions and discussions should be documented in:

- **GitHub Issues** for bugs and features
- **GitHub Discussions** for community topics
- **Meeting notes** for important decisions
- **README updates** for process changes
